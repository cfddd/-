`mutex()`方法内部
```java

  // Can't be initialized in the constructor because mocks don't call the constructor.
  private volatile Object mutexDoNotUseDirectly;

  private Object mutex() {
    Object mutex = mutexDoNotUseDirectly;
    if (mutex == null) {
      synchronized (this) {
        mutex = mutexDoNotUseDirectly;
        if (mutex == null) {
          mutexDoNotUseDirectly = mutex = new Object();
        }
      }
    }
    return mutex;
  }
```
这里就是多线程创建单实例的**懒汉模式**，但是他有一些不一样的地方，创建了一个本地变量mutex，是什么作用呢？

这是因为：每次读取 volatile 变量都有性能开销，需要从主内存读取，不能使用CPU缓存；需要内存屏障（memory barrier）来保证可见性，所以导致 比普通变量读取慢很多

**如果直接使用 mutexDoNotUseDirectly** ，有3次 volatile 读取，性能比使用本地变量objec一次读取开销大。
```java
// 不好的实现方式：
private Object mutex() {
    if (mutexDoNotUseDirectly == null) {        // 第1次 volatile 读取
        synchronized (this) {
            if (mutexDoNotUseDirectly == null) { // 第2次 volatile 读取
                mutexDoNotUseDirectly = new Object(); // volatile 写入
            }
        }
    }
    return mutexDoNotUseDirectly;               // 第3次 volatile 读取！
}
```
> 单线程情况下，编译时会自动优化volatile变量，两种方法显然不使用本地变量的更快

```java
  /**
   * Updates {@code storedPermits} and {@code nextFreeTicketMicros} based on the current time.
   */
  void resync(long nowMicros) {
    // if nextFreeTicket is in the past, resync to now
    if (nowMicros > nextFreeTicketMicros) {
      double newPermits = (nowMicros - nextFreeTicketMicros) / coolDownIntervalMicros();
      storedPermits = min(maxPermits, storedPermits + newPermits);
      nextFreeTicketMicros = nowMicros;
    }
  }
```
第一次新建会先执行 `resync` ,将：
- storedPermits = 0.0 (保持不变)
- nextFreeTicketMicros = nowMicros (更新为当前时间)

> `double newPermits = (nowMicros - nextFreeTicketMicros) / coolDownIntervalMicros();`coolDownIntervalMicros()未赋值，默认是0.0，这样触发不会报错，从IEEE 754浮点数标准理解，他是一个代表无限大的POSITIVE_INFINITY，但是后面有一个min方法保护又变成了0.0。详情参考 `java.lang.Double`

## SmoothBursty
有一个核心属性（剩余继承自 `SmoothRateLimiter`）
- maxBurstSeconds：代表流量限制的单位，也代表最大存活时间，默认1s
```java
  
@Override  
void doSetRate(double permitsPerSecond, double stableIntervalMicros) {  
  double oldMaxPermits = this.maxPermits;  
  maxPermits = maxBurstSeconds * permitsPerSecond;  
  if (oldMaxPermits == Double.POSITIVE_INFINITY) {  
    // if we don't special-case this, we would get storedPermits == NaN, below  
    storedPermits = maxPermits;  
  } else {  
    storedPermits =  
        (oldMaxPermits == 0.0)  
            ? 0.0 // initial state  
            : storedPermits * maxPermits / oldMaxPermits;  
  }  
}
```
先读取之前旧的 `maxPermits` ，然后设置新的。接着设置storedPermits才是重头戏：

**新的storedPermits = 旧的storedPermits × (新的maxPermits / 旧的maxPermits)**

可以看到他保持令牌比例的一致性，当限流速率改变时，令牌桶的容量也会改变，需要按比例调整当前存储的令牌数。

通过这个公式，RateLimiter在动态调整速率时能够平滑过渡，既不会突然释放大量令牌造成流量冲击，也不会突然清空令牌造成不必要的等待。

执行完成之后：
- storedPermits = 0.0 (保持不变)
- maxPermits = maxBurstSeconds * permitsPerSecond(更新为当前时间)

但是现在还出现了一个问题？storedPermits按照比例改变后，怎么释放呢？这个后面在解释，我么先接触创建新的限流器逻辑，回到rsync与每次调用限流的逻辑

### 每次调用限流的rsync逻辑

### acquire
从此 RateLimiter 获取给定数量的许可，并阻止直到可以授予请求。告诉睡眠时间（如果有）。

```java
@CanIgnoreReturnValue  
public double acquire(int permits) {  
  long microsToWait = reserve(permits);  
  stopwatch.sleepMicrosUninterruptibly(microsToWait);  
  return 1.0 * microsToWait / SECONDS.toMicros(1L);  
}
```
首先是最重要的 `reserve` 方法，它用来获得当前线程在限流初需要等待的时间
```java
final long reserve(int permits) {  
  checkPermits(permits);  
  synchronized (mutex()) {  
    return reserveAndGetWaitLength(permits, stopwatch.readMicros());  
  }

```
使用和前面提到相同的互斥锁

> 大量使用 `synchronized` 互斥，会不会对性能有影响？jvm对`synchronized`自带有优化，参考synchronized锁阶段

```java
  
@Override  
final long reserveEarliestAvailable(int requiredPermits, long nowMicros) {  
  resync(nowMicros);  
  long returnValue = nextFreeTicketMicros;  
  double storedPermitsToSpend = min(requiredPermits, this.storedPermits);  
  double freshPermits = requiredPermits - storedPermitsToSpend;  
  long waitMicros =  
      storedPermitsToWaitTime(this.storedPermits, storedPermitsToSpend)  
          + (long) (freshPermits * stableIntervalMicros);  
  
  this.nextFreeTicketMicros = LongMath.saturatedAdd(nextFreeTicketMicros, waitMicros);  
  this.storedPermits -= storedPermitsToSpend;  
  return returnValue;  
}
```
第一步，调用 `resync` ，可以理解，每次调用都需要先检查当前时间是否已经过了`nextFreeTicketMicros`，然后更新`storedPermits`和`nextFreeTicketMicros`，此时

- storedPermits = min(maxPermits, storedPermits + newPermits) 即 MIN(最大Permits，现有Permits + 本次需要的Permits)
- nextFreeTicketMicros = 当前时间

> 在这里就可以解决前面提到的问题了，每次限流前都会先执行 `resync` ，他会自动根据：
> 
> (本次计时时间 - 上次计时时间)/每个Permits时间间隔，计算出因为过去的时间而需要新增的Permits，然后自动更新storedPermits。
> 
> 这是一种类似懒加载的形式，不需要代码在显式调用释放Permits，所有逻辑都融合在`resync`计算过程中，非常优美的数学公式

第二步，计算需要使用的Permits
- storedPermitsToSpend 含义：从令牌桶中取出的令牌数量
- freshPermits 含义：需要等待新生成的令牌数量
- waitMicros含义：这次获取Permits需要等待的时间，SmoothBursty为 0L

第三步，扣减storedPermits，根据上面计算出的需要从令牌桶中取出的令牌数量`storedPermitsToSpend`。最后返回计算前的`nextFreeTicketMicros`作为最近可用时间

如果 最近可用时间 - 现在时间大于0 则需要等待，否则不需要

### 限流等待
```java
@CanIgnoreReturnValue  
public double acquire(int permits) {  
  long microsToWait = reserve(permits);  
  stopwatch.sleepMicrosUninterruptibly(microsToWait);  
  return 1.0 * microsToWait / SECONDS.toMicros(1L);  
}
```
根据计算出来的 `microsToWait`，调用`sleepMicrosUninterruptibly`方法让当前线程睡眠一段时间
