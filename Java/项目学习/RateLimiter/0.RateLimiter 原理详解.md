## Google Guava RateLimiter 原理详解

`com.google.common.util.concurrent.RateLimiter` 是 Google Guava 库中的一个强大的限流工具，它基于**令牌桶算法（Token Bucket Algorithm）**实现。让我详细为您解释其工作原理：

### 1. 令牌桶算法核心概念

**令牌桶算法**是一种经典的流量控制算法，其基本思想是：

- 🪣 **令牌桶**：想象有一个固定容量的桶
- 🎫 **令牌生成**：系统以恒定速率向桶中投放令牌
- 🚀 **请求处理**：每个请求需要消耗一个或多个令牌才能被处理
- ⏰ **等待机制**：如果桶中没有足够令牌，请求需要等待

### 2. RateLimiter 的两种实现
![](http://file.cfd.hhblog.top//myPicture/202509282010878.png)

Guava 提供了两种 RateLimiter 实现：

#### SmoothBursty（平滑突发型）
- 允许突发流量，可以"预支"未来的令牌
- 适合处理突发请求场景

#### SmoothWarmingUp（平滑预热型）  
- 有预热期，逐渐达到设定的速率
- 适合系统启动或长时间空闲后的场景

### 3. 核心工作机制

```java
// 创建限流器：每秒允许 5 个请求
RateLimiter rateLimiter = RateLimiter.create(5.0);

// 获取令牌（阻塞式）
rateLimiter.acquire();        // 获取 1 个令牌
rateLimiter.acquire(3);       // 获取 3 个令牌

// 尝试获取令牌（非阻塞式）
boolean success = rateLimiter.tryAcquire();
boolean success = rateLimiter.tryAcquire(2, 100, TimeUnit.MILLISECONDS);
```

### 4. 关键内部机制

#### 时间窗口计算
- 使用 `nextFreeTicketMicros` 记录下一个可用令牌的时间
- 通过时间差计算应该生成多少新令牌

#### 令牌存储
- `storedPermits` 存储当前可用的令牌数
- 有最大存储限制，防止无限累积

#### 预支机制
- 允许"借用"未来的令牌，但会影响后续请求的等待时间
- 这使得系统能够处理合理的突发流量

### 5. 实际应用场景

1. **API 限流**：防止接口被恶意调用
2. **资源保护**：保护数据库、缓存等资源不被过载
3. **流量整形**：将不规则的请求流量整形为平滑的输出
4. **系统保护**：防止系统因突发流量而崩溃

### 6. 优势特点

- ✅ **线程安全**：可以在多线程环境中安全使用
- ✅ **高性能**：基于 CAS 操作，避免锁竞争
- ✅ **灵活配置**：支持动态调整限流速率
- ✅ **突发处理**：合理处理短时间内的流量突发
- ✅ **预热机制**：避免系统冷启动时的性能问题

RateLimiter 通过这些精心设计的机制，为 Java 应用提供了一个既高效又灵活的限流解决方案，是构建高可用系统的重要工具。
