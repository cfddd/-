
## SSL 连接的理论持续时间

### 1. 理论上可以保持多久

```
理论上：SSL连接可以保持无限长时间
前提：连接两端都保持活跃状态
```

但实际上，多个层面都有超时机制：

## 各层面的超时机制

### 1. TCP 层面（操作系统层）

```
TCP Keep-Alive 机制：
├── 默认间隔: 2小时（Linux）
├── 探测次数: 9次
└── 总超时时间: 约 2小时 + (9 × 探测间隔)

如果2小时内无数据传输：
→ 操作系统发送 Keep-Alive 探测包
→ 如果服务器无响应 → 连接被关闭
```

### 2. SSL/TLS 层面（协议层）

```
SSL Session 超时：
├── Session ID 有效期: 通常 24小时
├── Session Ticket 有效期: 通常 24小时
└── 如果超时 → 需要重新握手

但连接本身不会因为Session超时而关闭
只是下次通信时需要重新握手
```

### 3. LDAP 应用层（应用层）

```
LDAP 服务器超时设置（常见配置）：
├── 空闲超时 (idle timeout): 5-30分钟
├── 最大连接时间: 1-2小时
└── 最大空闲时间: 15-60分钟

如果超过这些时间 → 服务器主动关闭连接
```

### 4. JVM/Java 层面（Java层）

```
Java LDAP 客户端超时：
├── connectTimeout: 默认无限制（直到系统超时）
├── readTimeout: 默认无限制
└── 但实际受操作系统TCP超时影响
```

## 实际场景中的持续时间

### 场景1：活跃连接（有数据传输）

```
连接创建 → 数据传输 → 数据传输 → 数据传输 → ...
   ↑                                              ↑
   可以保持数小时甚至数天（如果持续有数据）
```

### 场景2：空闲连接（无数据传输）

```
连接创建 → 数据传输 → [空闲] → [空闲] → [空闲] → 被关闭
   ↑                                    ↑
   通常5-30分钟后被服务器关闭
```

### 场景3：LDAP SSL 连接的具体情况

```
LDAP SSL 连接生命周期：

1. 连接建立: 0ms
   ├── TCP 三次握手: ~10-50ms
   └── SSL 握手: ~50-200ms

2. 数据传输: 1-100ms
   └── modifyAttributes 操作

3. 空闲状态: [开始计时]
   ├── 5分钟: [连接仍然活跃]
   ├── 15分钟: [连接仍然活跃]
   ├── 30分钟: [可能被服务器关闭]
   └── 2小时: [很可能被操作系统关闭]

4. 连接关闭:
   ├── 客户端主动关闭: 立即
   ├── 服务器超时关闭: 5-30分钟
   └── 操作系统超时关闭: 2小时+
```

## 影响持续时间的因素

### 1. 服务器配置

```yaml
# LDAP服务器配置示例
ldap:
  connection:
    idle_timeout: 30分钟    # 空闲超时
    max_connection_time: 2小时  # 最大连接时间
    keep_alive: true        # 是否启用Keep-Alive
```

### 2. 操作系统配置

```bash
# Linux TCP Keep-Alive 配置
net.ipv4.tcp_keepalive_time = 7200      # 2小时
net.ipv4.tcp_keepalive_probes = 9       # 探测次数
net.ipv4.tcp_keepalive_intvl = 75       # 探测间隔（秒）
```

### 3. 网络环境

```
影响因素：
├── 网络稳定性: 不稳定网络 → 连接更快断开
├── 防火墙设置: 可能设置更短的超时
├── 负载均衡器: 可能有自己的超时设置
└── NAT设备: 可能影响连接保持
```

## 实际测试数据（参考）

### 测试1：空闲连接持续时间

```
测试环境: LDAP服务器 + SSL连接
测试方法: 创建连接后不关闭，观察何时断开

结果：
├── 最短: 5分钟（服务器配置较短超时）
├── 常见: 15-30分钟（大多数LDAP服务器默认）
├── 较长: 1-2小时（服务器配置较长超时）
└── 最长: 2小时+（受操作系统TCP超时限制）
```

### 测试2：活跃连接持续时间

```
测试环境: 持续有数据传输的SSL连接
测试方法: 定期发送心跳或查询

结果：
├── 可以保持: 数小时甚至数天
└── 前提: 有数据传输，不超过服务器最大连接时间
```

## 总结

SSL 连接的一般持续时间：

| 场景 | 持续时间 | 说明 |
|------|---------|------|
| **空闲连接** | **5-30分钟** | 大多数LDAP服务器的默认空闲超时 |
| **活跃连接** | **数小时-数天** | 如果有持续数据传输 |
| **操作系统超时** | **2小时+** | TCP Keep-Alive 机制 |
| **理论最长** | **无限** | 如果两端都保持活跃且无超时配置 |

**关键点**：
- 即使连接会在 5-30 分钟后被服务器关闭，在这期间资源一直被占用
- 批量操作时，多个未关闭的连接会快速累积，导致资源耗尽
- 因此，应该在使用完后立即关闭连接，而不是等待超时

这就是为什么必须主动关闭连接的原因：即使连接最终会超时关闭，等待期间也会造成资源浪费和潜在问题。
